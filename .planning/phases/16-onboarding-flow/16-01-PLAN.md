# Phase 16 — Plan 01: Onboarding Flow

## Objective
Add a first-launch onboarding experience that introduces new users to the app, lets them configure key preferences (weekly goal, theme), and guides them to their first workout. The onboarding uses the glassmorphic design system established in phases 13–15 for a premium first impression.

## Scope
- 6 tasks across 2 files (1 new screen + 1 modified entry point)
- Estimated: ~500 lines new code, ~30 lines modified
- No new dependencies — uses existing glass components, Animated API, and ScrollView paging
- 4 onboarding pages: Welcome → Features → Setup → Ready
- Single plan — tasks execute sequentially

## Execution Context
```
App.tsx                                — onboarding gate: conditional rendering based on settings flag
src/screens/OnboardingScreen.tsx       — NEW: full onboarding screen with horizontal pager
src/database/services.ts               — (read-only) getSetting / setSetting for onboarding flag
src/components/glass/*                 — (read-only) GradientBackground, GlassCard, GlassButton
src/theme/*                            — (read-only) colors, tokens, useThemeControl
```

## Tasks

### Task 1: Onboarding gate — database check + conditional rendering in App.tsx
**Priority: High | Files: `App.tsx`**

Wire up the onboarding flag so the app shows the onboarding screen on first launch and the main app on subsequent launches.

**How:**

1. **Import** `getSetting` and `setSetting` from `./src/database/services`
2. **Import** `OnboardingScreen` from `./src/screens/OnboardingScreen`
3. **Add state** in `App()`: `const [onboardingDone, setOnboardingDone] = useState(false)`
4. **Check flag** in `attemptInit()` after `initializeDatabase()`:
   ```typescript
   const done = getSetting('onboarding_completed', 'false');
   setOnboardingDone(done === 'true');
   ```
5. **Add completion callback**:
   ```typescript
   const handleOnboardingComplete = () => {
     setSetting('onboarding_completed', 'true');
     setOnboardingDone(true);
   };
   ```
6. **Conditional render** in the return statement — when `!onboardingDone`, render `OnboardingScreen` inside `ThemeProvider` instead of `AppContent`:
   ```typescript
   return (
     <ErrorBoundary>
       <ThemeProvider>
         {onboardingDone ? <AppContent /> : <OnboardingScreen onComplete={handleOnboardingComplete} />}
       </ThemeProvider>
     </ErrorBoundary>
   );
   ```
7. **Skip non-critical init** when onboarding isn't done — notification permissions, sync, quick actions, and widget update can wait until after onboarding completes. Move these calls so they only run when `onboardingDone` becomes true (or keep them in attemptInit but accept they run early — either approach works since they're fire-and-forget).

**Note:** `OnboardingScreen` doesn't exist yet — this task creates the import and gate. The screen is created in Task 2. Create a minimal placeholder (`export default function OnboardingScreen...`) if needed for TypeScript to compile between tasks.

**Checkpoint:** App compiles. On first launch (or after clearing `onboarding_completed` setting), app shows the onboarding screen placeholder instead of the main tab navigator. On subsequent launches, app shows the normal main app.

### Task 2: OnboardingScreen — pager skeleton with glassmorphic base
**Priority: High | Files: `src/screens/OnboardingScreen.tsx`**

Create the onboarding screen with a horizontal scrolling pager, page indicator dots, and navigation buttons — all styled with the glassmorphic design system.

**How:**

1. **Create** `src/screens/OnboardingScreen.tsx`
2. **Props**: `{ onComplete: () => void }`
3. **Constants**: `TOTAL_PAGES = 4`
4. **Layout structure**:
   ```typescript
   <GradientBackground>
     <ScrollView
       ref={scrollRef}
       horizontal
       pagingEnabled
       showsHorizontalScrollIndicator={false}
       onScroll={handleScroll}
       scrollEventThrottle={16}
     >
       {/* Page components rendered here */}
     </ScrollView>

     {/* Bottom controls: page dots + buttons */}
     <View style={styles.controls}>
       <View style={styles.dots}>
         {[0,1,2,3].map(i => (
           <View key={i} style={[styles.dot, currentPage === i && styles.dotActive]} />
         ))}
       </View>
       <View style={styles.buttons}>
         {currentPage < TOTAL_PAGES - 1 ? (
           <>
             <Pressable onPress={handleSkip}>
               <Text style={styles.skipText}>Skip</Text>
             </Pressable>
             <GlassButton title="Next" onPress={handleNext} icon="arrow-forward" />
           </>
         ) : null}
       </View>
     </View>
   </GradientBackground>
   ```
5. **Page tracking**: Use `onScroll` to calculate `currentPage` from `event.nativeEvent.contentOffset.x / screenWidth`
6. **Navigation helpers**:
   - `handleNext()`: scroll to next page via `scrollRef.current?.scrollTo({ x: (currentPage + 1) * screenWidth })`
   - `handleSkip()`: scroll directly to last page (setup page, not past it — let user see the setup)
7. **Screen dimensions**: `useWindowDimensions()` for responsive page width
8. **Glassmorphic styling**:
   - `GradientBackground` wrapper for the entire screen
   - Dots: inactive = `glassBorder` color, active = `primary` color, pill shape
   - Skip text: `textSecondary` color
   - Next button: `GlassButton` primary variant
   - Bottom controls area: positioned absolutely at bottom with padding
9. **SafeArea**: Use `paddingTop` to account for status bar (same pattern as other screens with custom headers — use a fixed `paddingTop: 56` or measure with `useSafeAreaInsets` if available)
10. **Styles**: Use `useMemo(() => createStyles(colors), [colors])` pattern matching existing screens

**Checkpoint:** OnboardingScreen renders with gradient background. Horizontal swiping between 4 empty pages works. Page indicator dots update on swipe. Next button scrolls to next page. Skip jumps ahead. Bottom controls are glassmorphic.

Build out the content for the first two onboarding pages: a welcome splash and a feature showcase.

### Task 4: OnboardingScreen — setup page
**Priority: High | Files: `src/screens/OnboardingScreen.tsx`**

Build the setup page where users configure their weekly goal and theme preference before starting.

**How:**

1. **Page 3 — Setup**:
   - Section title: **"Make it yours"** — top of page
   - **Weekly Goal** section:
     - Label: "Weekly workout goal"

2. **Imports needed**: `setSetting` from database services, `useThemeControl` for theme mode

3. **State management**:

**Checkpoint:** Page 3 shows weekly goal and theme selectors. Tapping a goal number highlights it and saves to database. Tapping a theme option switches the app theme immediately (gradient background updates live). Selections persist if user swipes back and forward.

### Task 5: OnboardingScreen — ready page + completion flow
**Priority: High | Files: `src/screens/OnboardingScreen.tsx`**

Build the final page with call-to-action buttons that complete onboarding and transition to the main app.

**How:**

1. **Page 4 — Ready**:
   - Large Ionicons icon: `checkmark-circle` (size ~80, `success` color) — centered
   - Title: **"You're all set!"** — large, bold
   - Subtitle: **"Start logging workouts and watch your progress grow."** — secondary text
   - Primary CTA: `GlassButton` primary, title **"Start Your First Workout"**, icon `barbell-outline`, size `lg`
   - Secondary CTA: `GlassButton` secondary, title **"Explore Dashboard"**, size `md`
   - Animated entrance: same fade+slide pattern
   - No Next/Skip buttons on this page (handled in Task 2's conditional rendering — last page hides the bottom Next/Skip bar)

2. **Completion handlers**:
   - `handleStartWorkout()`:
     1. Call `onComplete()` (sets flag + shows main app via App.tsx callback)
     2. After a brief delay (50ms setTimeout), navigate to the "Log Workout" tab — since `AppContent` will now be mounted with `NavigationContainer`, we need to use `navigationRef` from App.tsx. Pass an optional `initialRoute` hint via the `onComplete` callback, or use `navigationRef.navigate('Log Workout')` after mount.

     **Simplest approach**: Change `onComplete` prop to accept an optional route hint:
     ```typescript
     onComplete: (initialRoute?: string) => void
     ```
     In App.tsx, after setting `onboardingDone(true)`, if `initialRoute` is provided, use a `useEffect` or `setTimeout` to navigate:
     ```typescript
     const handleOnboardingComplete = (initialRoute?: string) => {
       setSetting('onboarding_completed', 'true');
       setOnboardingDone(true);
       if (initialRoute) {
         setTimeout(() => {
           navigationRef.current?.navigate(initialRoute as never);
         }, 100);
       }
     };
     ```
   - `handleExploreDashboard()`: Call `onComplete()` with no route hint — Dashboard is the default tab

3. **Layout**: CTAs stacked vertically below the title/subtitle with spacing between them

**Checkpoint:** Page 4 shows celebration icon, title, and two CTA buttons. "Start Your First Workout" completes onboarding and lands on the Log Workout tab. "Explore Dashboard" completes onboarding and lands on Dashboard. Subsequent app launches skip onboarding entirely.

## Notes
- **No new dependencies**: Uses ScrollView `pagingEnabled` for paging, React Native `Animated` for entrance animations, and existing glass components for styling. No need for react-native-reanimated, lottie, or a pager library.
- **Notification permissions**: Already requested in App.tsx `attemptInit()`. The onboarding doesn't add a separate notification permission prompt — the OS prompt fires on first launch regardless. The setup page can include a reminders toggle but the actual permission is handled by the existing flow.
- **Settings persistence**: Weekly goal and theme are saved immediately on selection (same pattern as SettingsScreen). If user doesn't interact with the setup page, defaults apply (goal=3, theme=system).
- **Re-triggering onboarding**: To re-show onboarding during development, clear the `onboarding_completed` setting from the database or uninstall the app.
