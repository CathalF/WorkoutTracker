### Task 2: Fix TypeScript errors in quickActionHandler.ts
**Priority: High | Files: `src/utils/quickActionHandler.ts`**

There are 3 TypeScript errors caused by `navigationRef.navigate('Log Workout' as never, {...} as never)` — the `as never` cast pattern doesn't satisfy the typed `navigate()` overloads properly.

**How:**

The root cause is that `createNavigationContainerRef()` in App.tsx is untyped — it creates a `NavigationContainerRef<ReactNavigation.RootParamList>` which defaults to `Record<string, undefined>`. The tab navigator's param list isn't connected to the ref.

1. **Define `RootTabParamList`** in `src/navigation/AppNavigator.tsx` (or a shared types file) — but the simplest fix is to type the `navigationRef` in App.tsx:

   In `App.tsx`, change:
   ```typescript
   export const navigationRef = createNavigationContainerRef();
   ```
   to:
   ```typescript
   export const navigationRef = createNavigationContainerRef<RootTabParamList>();
   ```

   Where `RootTabParamList` is defined (in App.tsx or imported):
   ```typescript
   type RootTabParamList = {
     Dashboard: undefined;
     'Log Workout': { screen?: string; params?: Record<string, unknown> };
     History: undefined;
     Progress: undefined;
   };
   ```

2. **Update `quickActionHandler.ts`** — remove all `as never` casts. The typed `navigationRef` should accept the tab route names directly. If nested navigation params still cause type issues, use a minimal cast (`as any` on the nested params object only) rather than casting the entire call.

3. Run `npx tsc --noEmit` — should show zero errors.

**Checkpoint:** Zero TypeScript errors. Quick action handler navigates correctly with proper types.

### Task 3: Theme-aware loading and error states in App.tsx
**Priority: Medium | Files: `App.tsx`**

The loading spinner and error state in `App()` use hardcoded colors (`#F2F2F7`, `#007AFF`, `#000`, `#FF3B30`). These don't match the glassmorphic theme and look jarring in dark mode. Since these render BEFORE `ThemeProvider` mounts, we need to use the system color scheme directly.

**How:**

1. Move the `useColorScheme()` call from `AppContent` up to the `App` component level (or add a second call in `App`)
2. Create a minimal color lookup based on system scheme:
   ```typescript
   const scheme = useColorScheme();
   const isDark = scheme === 'dark';
   ```
3. Update the loading state:
   - Background: `isDark ? '#000000' : '#F2F2F7'`
   - ActivityIndicator color: `isDark ? '#0A84FF' : '#007AFF'`
4. Update the error state:
   - Background: same as loading
   - Error title color: `isDark ? '#FFFFFF' : '#000000'`
   - Error subtitle color: `'#8E8E93'` (same in both themes)
   - Retry button background: `isDark ? '#0A84FF' : '#007AFF'`
   - Icon color stays `#FF3B30` (red is the same in both themes, close enough)

5. Update the `styles` from `StyleSheet.create` to inline styles or a `useMemo`-based stylesheet that uses the scheme values.

**Checkpoint:** Loading spinner and error screen look correct in both light and dark system themes. Colors match the app's theme palette.
