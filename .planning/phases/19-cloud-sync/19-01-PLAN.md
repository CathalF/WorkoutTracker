# Phase 19-01 — Cloud Sync & Data Migration

## Objective

Add bidirectional sync between local SQLite and Supabase Postgres. Existing local workout data migrates to the cloud on first sync. Subsequent syncs push local changes and pull remote changes using last-write-wins conflict resolution. Local SQLite remains the primary data source — Supabase is the cloud backup and future foundation for social features.

## Context & Decisions

- **Custom sync over third-party** — no new dependencies; Supastash too immature, PowerSync/WatermelonDB overkill for single-user
- **Last-write-wins (LWW)** — sufficient for single-user app; compare `updated_at` timestamps
- **Local SQLite stays primary** — all reads come from SQLite; Supabase is push/pull backup
- **ID mapping table** — bridges local INTEGER autoincrement IDs to remote UUIDs
- **Sync order respects FK constraints** — muscle_groups → exercises → programs → templates → template_exercises → workouts → sets → personal_records → settings
- **`updated_at` columns added to local SQLite** — enables change tracking without a separate changelog
- **Auto-sync on foreground** — 5-minute cooldown; manual "Sync Now" always available
- **Initial migration is a one-time bulk push** — all existing local data pushed to Supabase on first authenticated sync

### 4. Create sync engine — `src/services/syncEngine.ts`

The core sync service with push and pull logic:

**Exports:**
```typescript
interface SyncResult {
  success: boolean;
  pushed: number;     // rows pushed to Supabase
  pulled: number;     // rows pulled from Supabase
  errors: string[];   // human-readable error messages
  duration: number;   // ms
}

function syncAll(userId: string): Promise<SyncResult>
function initialMigration(userId: string): Promise<SyncResult>
```

**`initialMigration(userId)` — First-time bulk push:**
1. Fetch Supabase `muscle_groups` → build name→id mapping (these are shared reference data, don't push)
2. Map local `exercises` seed data by name+muscle_group to Supabase seed exercises → store ID mappings
3. Push custom exercises to Supabase → store ID mappings
4. Push `programs` → store ID mappings
5. Push `workout_templates` (map muscle_group_id, program_id via id_map) → store ID mappings
6. Push `template_exercises` (map template_id, exercise_id via id_map) → store ID mappings
7. Push `workouts` (map muscle_group_id via id_map) → store ID mappings
8. Push `sets` (map workout_id, exercise_id via id_map) → store ID mappings
9. Push `personal_records` (map exercise_id via id_map) → store ID mappings
10. Push `user_settings` (key-value pairs) → store ID mappings
11. Set `last_synced_at` to server time
12. Mark migration complete

**`syncAll(userId)` — Incremental sync:**
1. Get `last_synced_at` from sync_meta
2. **PUSH phase** — for each table in FK order:
   - Query local rows where `updated_at > last_synced_at` (includes soft-deleted rows)
   - Map local IDs to remote UUIDs via id_map
   - Upsert to Supabase (new rows get INSERT, existing get UPDATE)
   - Store any new ID mappings
3. **PULL phase** — for each table in FK order:
   - Query Supabase for rows where `updated_at > last_synced_at`
   - For each remote row:
     - If exists in id_map → compare timestamps, update local if remote is newer
     - If not in id_map → insert locally, create ID mapping
   - Handle soft deletes: if remote `deleted_at` is set, soft-delete locally
4. Update `last_synced_at` to current server time

**Key implementation details:**
- Use `supabase.from('table').upsert()` for push (handles insert + update)
- Use `supabase.from('table').select().gt('updated_at', lastSyncedAt)` for pull
- Wrap each table sync in try/catch — continue with next table on failure
- Return aggregate `SyncResult` with counts and errors

**Checkpoint:** `initialMigration()` pushes all local data to Supabase. `syncAll()` pushes changes and pulls remote changes. ID mappings are correct.

### 5. Create SyncContext — `src/contexts/SyncContext.tsx`

React context to manage sync state and provide sync functions to the UI:

```typescript
interface SyncContextValue {
  isSyncing: boolean;
  lastSyncedAt: string | null;   // ISO timestamp
  lastSyncResult: SyncResult | null;
  syncNow: () => Promise<void>;  // manual trigger
  hasMigrated: boolean;
}
```

**Implementation:**
- On mount: check if initial migration is needed (`hasMigrated()`)
- If not migrated and user is authenticated → run `initialMigration(userId)` automatically
- After migration: run `syncAll()` to pull any server-side data
- Auto-sync: listen to AppState — when app returns to foreground, sync if >5 minutes since last sync
- Expose `syncNow()` for manual trigger from Settings

**Provider placement:** Wrap inside `AuthProvider` in `App.tsx` (needs auth user for sync).

**Checkpoint:** SyncProvider mounts. Initial migration runs on first auth. Auto-sync triggers on foreground.
