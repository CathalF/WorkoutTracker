### Task 1: Fix UAT-002 & UAT-003: ExercisePicker param forwarding drops ActiveWorkout params

<task type="auto">
  <name>Fix ExercisePicker navigation param loss (UAT-002 + UAT-003 root cause)</name>
  <files>
    src/screens/ExercisePickerScreen.tsx
    src/screens/ActiveWorkoutScreen.tsx
  </files>
  <action>
**Root cause:** `ExercisePickerScreen.selectExercise()` at line 84 calls:
```typescript
navigation.navigate('ActiveWorkout', {
  ...route.params,   // ExercisePicker params — missing muscleGroupId, splitLabel
  selectedExercise: { id, name },
} as any);
```

The spread `...route.params` contains ExercisePicker's own params (`workoutId`, `muscleGroupIds`, `alreadyAddedIds`) but NOT `muscleGroupId` (singular) or `splitLabel` — those are ActiveWorkout params that were never passed to ExercisePicker. The `as any` cast silences the TypeScript error. Even though React Navigation merges params for existing stack screens, this introduces ExercisePicker-only keys and may cause issues on some platforms/versions.

**Fix 1 — ExercisePickerScreen.tsx:**
Change `selectExercise` to pass ONLY `selectedExercise`. Remove the `...route.params` spread and `as any` cast. React Navigation's `navigate()` to an existing stack screen automatically merges the new params with existing params, so `muscleGroupId`, `splitLabel`, etc. are preserved:

```typescript
const selectExercise = (exerciseId: number, exerciseName: string) => {
  if (alreadyAddedSet.has(exerciseId)) return;
  navigation.navigate('ActiveWorkout', {
    selectedExercise: { id: exerciseId, name: exerciseName },
  } as any);
};
```

**Fix 2 — ActiveWorkoutScreen.tsx:**
Defense-in-depth: save `muscleGroupId`, `splitLabel`, and `muscleGroupIds` to refs on first mount so they survive any param changes:

After line 53 (`lastSelectedRef`), add:
```typescript
const origMuscleGroupIdRef = useRef(muscleGroupId);
const origSplitLabelRef = useRef(splitLabel);
const origMuscleGroupIdsRef = useRef(muscleGroupIds);
```

In `handleFinishWorkout` (line 440), change:
```typescript
const resolvedMuscleGroupId =
  muscleGroupId || origMuscleGroupIdRef.current || getExerciseMuscleGroupId(exercises[0].exerciseId);
```

In `handleSaveTemplate` (line 500), change:
```typescript
const resolvedMgId = muscleGroupId || origMuscleGroupIdRef.current || getExerciseMuscleGroupId(exercises[0].exerciseId);
const resolvedSplitLabel = splitLabel || origSplitLabelRef.current || 'Custom';
const resolvedMgIds = muscleGroupIds?.length > 0 ? muscleGroupIds : origMuscleGroupIdsRef.current;
createTemplate(name, resolvedMgId, resolvedSplitLabel, resolvedMgIds, templateExercises);
```

### Task 2: Fix UAT-004: PR detection returns false on first-ever exercise set

<task type="auto">
  <name>Fix PR detection for first-time exercises (UAT-004)</name>
  <files>
    src/database/services.ts
  </files>
  <action>
**Root cause:** `checkForWeightPR()` (line 498) and `checkForRepsPR()` (line 509) return `false` when no prior sets exist in the database (`MAX(weight)` / `MAX(reps)` returns `NULL`). This means the very first set logged for an exercise is never detected as a PR. Since PR detection runs during the workout (before sets are saved to DB), and the first workout's sets ARE the first data in the table, PRs are never recorded. The `personal_records` table stays empty, so Dashboard shows no PR cards.

**Fix — services.ts:**

In `checkForWeightPR` (line 498), change:
```typescript
if (row?.max_weight === null || row?.max_weight === undefined) return false;
```
to:
```typescript
if (row?.max_weight === null || row?.max_weight === undefined) return true;
```

In `checkForRepsPR` (line 509), change:
```typescript
if (row?.max_reps === null || row?.max_reps === undefined) return false;
```
to:
```typescript
if (row?.max_reps === null || row?.max_reps === undefined) return true;
```

First-time sets for any exercise are always PRs — this is the correct behavior.
  </action>
  <verify>
1. Start a fresh workout, pick an exercise that has never been logged before
2. Log a set with valid weight and reps, tap the checkmark to complete it
3. A PR celebration toast should appear
4. Finish the workout
5. Go to Dashboard — the PR card section should now display the recorded PR
6. Start another workout with the same exercise, beat the previous weight → new PR detected
7. Same weight, more reps → new reps PR detected
  </verify>
  <done>
- First-time exercise sets correctly detected as PRs
- PR celebration toasts appear during workouts
- Dashboard PR cards section displays recorded PRs
- Subsequent PRs (new weight or rep records) also detected correctly
  </done>
</task>
