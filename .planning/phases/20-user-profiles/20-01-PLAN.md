# Phase 20-01 — User Profiles

## Objective

Add user profiles to the app: display name, bio, and photo avatar. Profile setup happens once after first authentication. Profiles are editable from Settings. Other users' profiles are viewable read-only.

## Execution Context

- **Phase context:** `.planning/phases/20-user-profiles/CONTEXT.md`
- **Supabase client:** `src/lib/supabase.ts` — already configured with auth
- **Auth context:** `src/contexts/AuthContext.tsx` — provides session/user
- **Sync context:** `src/contexts/SyncContext.tsx` — cloud sync engine
- **App entry:** `App.tsx` — AuthGate manages onboarding → auth → main app flow
- **Settings:** `src/screens/SettingsScreen.tsx` — gear icon from Dashboard
- **Navigation:** `src/navigation/DashboardStackNavigator.tsx` — DashboardHome + Settings
- **Design system:** `src/components/glass/` — GlassCard, GlassButton, GlassInput, GlassModal, GradientBackground
- **Theme:** `src/theme/` — useThemeControl, ThemeColors, tokens (typography, spacing, radii)
- **Types:** `src/types/index.ts` — app-wide type definitions

## Context & Decisions

- **Profile setup is post-auth** — Onboarding happens before auth (local preferences). Profile requires Supabase auth for avatar upload, so ProfileSetupScreen shows once after first sign-in when no profile exists.
- **Profiles live in Supabase only** — No local SQLite profiles table. Profiles are cloud-native since they're identity data visible to other users. Cached in React context during the session.
- **Supabase Storage for avatars** — `avatars` bucket with public read access. File path: `{user_id}/avatar.{ext}`. Public URL used as `avatar_url` in profiles table.
- **expo-image-picker for photo selection** — Camera roll picker. No camera capture needed (simpler, no extra permissions).
- **Read-only view of others** — `UserProfileScreen` accepts a `userId` param and fetches that user's profile. No social actions (Phase 21+).
- **RLS: own profile full access, others read-only** — Users can CRUD their own profile. All authenticated users can SELECT any profile (needed for social phases).
- **Glass design system** — All new screens use GradientBackground, GlassCard, GlassInput, GlassButton consistent with existing app.

### 2. Supabase schema — profiles table + avatars bucket

Write SQL reference file `src/lib/supabase-profiles.sql` for the user to run in their Supabase SQL editor.

**Profiles table:**
```sql
CREATE TABLE profiles (
  id UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  display_name TEXT NOT NULL DEFAULT '',
  bio TEXT NOT NULL DEFAULT '',
  avatar_url TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW() NOT NULL,
  updated_at TIMESTAMPTZ DEFAULT NOW() NOT NULL
);

CREATE INDEX idx_profiles_display_name ON profiles(display_name);

ALTER TABLE profiles ENABLE ROW LEVEL SECURITY;

-- Anyone authenticated can view any profile
CREATE POLICY "Profiles are viewable by authenticated users"
  ON profiles FOR SELECT
  USING (auth.role() = 'authenticated');

-- Users can insert their own profile
CREATE POLICY "Users can create own profile"
  ON profiles FOR INSERT
  WITH CHECK ((SELECT auth.uid()) = id);

-- Users can update their own profile
CREATE POLICY "Users can update own profile"
  ON profiles FOR UPDATE
  USING ((SELECT auth.uid()) = id)
  WITH CHECK ((SELECT auth.uid()) = id);

-- Auto-update updated_at
CREATE TRIGGER profiles_updated_at
  BEFORE UPDATE ON profiles
  FOR EACH ROW
  EXECUTE FUNCTION update_modified_column();
```

**Avatars storage bucket:**
```sql
INSERT INTO storage.buckets (id, name, public) VALUES ('avatars', 'avatars', true);

-- Anyone can view avatars (public bucket)
CREATE POLICY "Avatar images are publicly accessible"
  ON storage.objects FOR SELECT
  USING (bucket_id = 'avatars');

-- Users can upload their own avatar
CREATE POLICY "Users can upload own avatar"
  ON storage.objects FOR INSERT
  WITH CHECK (bucket_id = 'avatars' AND (SELECT auth.uid())::text = (storage.foldername(name))[1]);

-- Users can update their own avatar
CREATE POLICY "Users can update own avatar"
  ON storage.objects FOR UPDATE
  USING (bucket_id = 'avatars' AND (SELECT auth.uid())::text = (storage.foldername(name))[1]);

-- Users can delete their own avatar
CREATE POLICY "Users can delete own avatar"
  ON storage.objects FOR DELETE
  USING (bucket_id = 'avatars' AND (SELECT auth.uid())::text = (storage.foldername(name))[1]);
```

**Checkpoint:** SQL file exists with valid syntax. Tables and policies are defined.

### 4. Create profile service — `src/services/profileService.ts`

Service functions for profile CRUD and avatar upload:

**Functions:**
- `getProfile(userId: string): Promise<UserProfile | null>` — Fetch a user's profile by ID. Returns null if not found.
- `getMyProfile(): Promise<UserProfile | null>` — Convenience wrapper using current auth session.
- `upsertProfile(profile: { display_name: string; bio: string; avatar_url?: string | null }): Promise<UserProfile>` — Create or update the current user's profile. Uses upsert with `id = auth.uid()`.
- `uploadAvatar(userId: string, uri: string): Promise<string>` — Upload image from local URI to Supabase Storage. Returns the public URL. Steps:
  1. Read the file as a blob/arraybuffer from the local URI
  2. Determine file extension from URI or MIME type
  3. Upload to `avatars/{userId}/avatar.{ext}` (overwrite existing)
  4. Get public URL from Supabase Storage
  5. Return the public URL
- `deleteAvatar(userId: string): Promise<void>` — Remove avatar from storage.

**Key details:**
- Use `supabase.storage.from('avatars')` for file operations
- Use `supabase.from('profiles')` for table operations
- Avatar upload: fetch the local URI, convert to blob, upload with `contentType` header
- For React Native image upload: use `fetch(uri)` → `.blob()` → upload to Supabase Storage

**Checkpoint:** Service functions compile. Importing them doesn't crash.

### 5. Create ProfileContext — `src/contexts/ProfileContext.tsx`

React context that caches the current user's profile and provides update functions:

**Interface:**
```typescript
interface ProfileContextValue {
  profile: UserProfile | null;
  isLoading: boolean;
  hasProfile: boolean;       // true if profile exists in Supabase
  updateProfile: (data: { display_name: string; bio: string }) => Promise<void>;
  updateAvatar: (localUri: string) => Promise<void>;
  refreshProfile: () => Promise<void>;
}
```

**Implementation:**
- `ProfileProvider` wraps children, provides context
- On mount (when user is authenticated): call `getMyProfile()` to load profile
- `hasProfile` derived from whether profile was found (not null)
- `updateProfile` calls `upsertProfile()` and refreshes local state
- `updateAvatar` calls `uploadAvatar()` then `upsertProfile()` with new URL, refreshes state
- `refreshProfile` re-fetches from Supabase
- `isLoading` true during initial fetch
- Export `useProfile()` hook

**Checkpoint:** ProfileProvider renders children. `useProfile()` returns expected shape.

### 12. Create avatar picker utility — `src/utils/avatarPicker.ts`

Shared utility used by both ProfileSetupScreen and EditProfileScreen:

```typescript
export async function pickAvatar(): Promise<string | null> {
  const permission = await ImagePicker.requestMediaLibraryPermissionsAsync();
  if (!permission.granted) return null;

  const result = await ImagePicker.launchImageLibraryAsync({
    mediaTypes: 'images',
    allowsEditing: true,
    aspect: [1, 1],
    quality: 0.7,
  });

  if (result.canceled) return null;
  return result.assets[0].uri;
}
```

**Checkpoint:** Function compiles. Returns URI string or null.
