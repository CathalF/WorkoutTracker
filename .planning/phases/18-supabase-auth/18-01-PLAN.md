# Phase 18-01 ā€” Supabase Setup & Authentication

## Objective

Add Supabase authentication to the app. Users must sign in (email/password) before accessing the main app. The app continues using local SQLite for all workout data ā€” cloud sync comes in Phase 19.

## Execution Context

- **Research:** `.planning/phases/18-supabase-auth/RESEARCH.md`
- **Current app entry:** `App.tsx` ā€” manages DB init, onboarding gate, renders `AppContent` or `OnboardingScreen`
- **Navigation:** `src/navigation/AppNavigator.tsx` ā€” bottom tabs (Dashboard, Log Workout, History, Progress)
- **Design system:** `src/components/glass/` ā€” GlassCard, GlassButton, GlassInput, GlassModal, GradientBackground
- **Theme:** `src/theme/` ā€” ThemeProvider, useTheme, useThemeControl, dark/light colors, typography/spacing/radii tokens
- **Settings:** `src/screens/SettingsScreen.tsx` ā€” gear icon from Dashboard, uses GradientBackground

## Context & Decisions

- **Email/password only** ā€” no social login (Apple/Google deferred to later phase)
- **AsyncStorage for auth persistence** ā€” more battle-tested than expo-sqlite localStorage for session storage
- **App continues using local SQLite** ā€” auth is a separate gate; no data migration in this phase
- **Auth flow:** First launch ā†’ Onboarding ā†’ Auth ā†’ Main app. Returning ā†’ auto-restore session ā†’ Main app
- **Supabase project setup is manual** ā€” user creates project in Supabase Dashboard; we provide SQL schema and RLS policies as reference
- **`detectSessionInUrl: false`** ā€” required for React Native; prevents OAuth redirect interference
- **AppState listener for token refresh** ā€” ensures tokens refresh when app returns to foreground
- **`EXPO_PUBLIC_` prefix** for env vars ā€” Expo strips unprefixed vars from client bundle
- **Anon key is safe to expose** ā€” RLS protects data server-side

### 4. Create AuthContext ā€” `src/contexts/AuthContext.tsx`
Create a React context provider that manages auth state:

**Interface:**
```typescript
interface AuthContextValue {
  session: Session | null;
  user: User | null;
  isLoading: boolean;
  signIn: (email: string, password: string) => Promise<{ error: AuthError | null }>;
  signUp: (email: string, password: string) => Promise<{ error: AuthError | null }>;
  signOut: () => Promise<void>;
  resetPassword: (email: string) => Promise<{ error: AuthError | null }>;
}
```

**Implementation:**
- `AuthProvider` wraps children, provides context value
- On mount: call `supabase.auth.getSession()` to restore existing session
- Subscribe to `supabase.auth.onAuthStateChange()` ā€” update session/user on SIGNED_IN, SIGNED_OUT, TOKEN_REFRESHED
- Unsubscribe from auth listener on unmount (prevent memory leak)
- `isLoading` is true until initial session check completes
- `signIn`, `signUp`, `signOut`, `resetPassword` are thin wrappers around `supabase.auth.*`
- Export `useAuth()` hook for consumers

**Checkpoint:** AuthProvider renders children. `useAuth()` returns `{ session: null, user: null, isLoading: false }` when no session exists.

### 9. Update App.tsx ā€” Add auth gating

Modify the app's init flow to include auth:

**Current flow:**
```
DB init ā†’ onboarding check ā†’ OnboardingScreen | AppContent
```

**New flow:**
```
DB init ā†’ AuthProvider wraps everything
  ā†’ isLoading? ā†’ Loading spinner
  ā†’ No session? ā†’ onboarding check ā†’ OnboardingScreen (if needed) ā†’ AuthStackNavigator
  ā†’ Has session? ā†’ onboarding check ā†’ OnboardingScreen (if needed) ā†’ AppContent
```

**Specific changes:**
- Import `AuthProvider` and `useAuth` from `src/contexts/AuthContext`
- Wrap `ThemeProvider` children with `AuthProvider`
- Create a new `AuthGate` component inside `App()` that reads `useAuth()`:
  - If `isLoading`, show loading spinner (reuse existing loading UI)
  - If no session AND onboarding not done, show OnboardingScreen with modified `onComplete` that goes to auth
  - If no session AND onboarding done, show AuthStackNavigator
  - If session exists AND onboarding not done, show OnboardingScreen
  - If session exists AND onboarding done, show AppContent
- The key insight: onboarding is local setup (theme, weekly goal), auth is cloud identity. Onboarding should happen first, then auth.

**Simplified gate logic:**
```
if (isLoading) ā†’ spinner
if (!onboardingDone) ā†’ OnboardingScreen (onComplete sets onboardingDone)
if (!session) ā†’ AuthStackNavigator
else ā†’ AppContent
```

**Checkpoint:** App loads. Without Supabase credentials, shows auth screens after onboarding. Can navigate between auth screens.

### 11. Create Supabase SQL reference ā€” `src/lib/supabase-schema.sql`

Write the Postgres schema + RLS policies as a reference file for the user to run in their Supabase SQL editor. This is NOT executed by the app ā€” it's a developer setup guide.

**Contents:**
- CREATE TABLE statements mirroring local SQLite schema but with:
  - `id UUID DEFAULT gen_random_uuid() PRIMARY KEY`
  - `user_id UUID REFERENCES auth.users(id) NOT NULL`
  - `created_at TIMESTAMPTZ DEFAULT NOW()`
  - `updated_at TIMESTAMPTZ DEFAULT NOW()`
  - `deleted_at TIMESTAMPTZ` (soft delete for future sync)
- Tables: muscle_groups (no user_id ā€” shared), exercises (conditional user_id for custom), workouts, sets, programs, workout_templates, template_exercises, personal_records, settings
- ENABLE ROW LEVEL SECURITY on all tables
- RLS policies (SELECT/INSERT/UPDATE/DELETE) with `(SELECT auth.uid()) = user_id`
- Indexes on all `user_id` columns
- `update_modified_column()` trigger function + triggers on each table
- Seed data insertion for muscle_groups and default exercises

**Checkpoint:** SQL file is syntactically valid. Tables, policies, indexes, and triggers are defined for all tables.
